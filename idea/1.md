# 风控项目管理平台 — 完整方案构想

---

## 一、背景与痛点

当前风控团队的需求管理依赖钉钉文档，存在以下核心问题：

| 痛点         | 具体表现                           |
|------------|--------------------------------|
| **状态不透明**  | 文档散落在多个群聊和文件夹中，无法一眼看到所有需求的当前状态 |
| **流转无规则**  | 需求从"想法"到"上线"的路径不清晰，依赖口头沟通推进    |
| **人机协作割裂** | AI Agent 的处理结果需要人工复制回文档，无法形成闭环 |
| **依赖关系隐性** | 任务之间的阻塞关系只存在于人脑中，没有显式表达        |
| **复盘困难**   | 缺少时间线和过程记录，事后无法追溯一个需求经历了什么     |

这个平台的目标是：**把风控团队的需求全生命周期，从"脑子里的想法"到"线上运行的规则"，变成一条可追踪、可自动化、可度量的流水线。
**

## 二、核心理念

### 2.1 三层架构

```
┌─────────────────────────────────────────────┐
│           看板层 (Board Layer)                │
│   可视化任务状态 · 拖拽流转 · 筛选视图        │
├─────────────────────────────────────────────┤
│           工作流层 (Workflow Layer)           │
│   状态机引擎 · 自动化规则 · 条件触发          │
├─────────────────────────────────────────────┤
│           Agent 层 (Agent Layer)             │
│   AI Agent · 人工节点 · 混合执行              │
└─────────────────────────────────────────────┘
```

- **看板层**面向人：PM、开发者、审核者通过看板交互
- **工作流层**面向规则：定义"什么条件下，任务从 A 状态流转到 B 状态"
- **Agent 层**面向执行：AI 和人都是"执行者"，平台统一调度

### 2.2 核心原则

1. **状态即真相** — 任务在哪里，就是什么阶段，不需要额外问人
2. **流转即事件** — 每次状态变更都是一次可审计的事件
3. **Agent 即队友** — AI Agent 和人类在同一个看板上协作，没有区别对待
4. **依赖即阻塞** — 未解除的依赖会自动阻止下游任务推进

## 三、任务状态机（完整定义）

### 3.1 状态全景

```
                        ┌──────────────┐
                        │  ① 想法待完善  │ ◄── PM 提交初始需求
                        │  (Draft)      │
                        └──────┬───────┘
                               │ 内部评审通过
                               ▼
              ┌──► ② 想法 (Idea) ◄──┐
              │    (已确认的idea)    │  PM 需要继续完善
              │                     │──────────────────┐
              │  拆解为任务          │                  │
              └─────────────────────┘                  │
                               │                       │
                               ▼                       │
                        ③ Task-TODO                    │
                        (待执行)                       │
                               │                       │
                    分配给 Agent / 人工                 │
                               │                       │
                               ▼                       │
                   ④ Task-In-Progress                  │
                   ┌───────┴────────┐                  │
                   ▼                ▼                  │
            ④-a In-AI        ④-b In-Human             │
           (AI Agent执行)     (人工执行)                │
                   │                │                  │
                   └───────┬────────┘                  │
                           │ 执行完成                   │
                           ▼                           │
                   ⑤ Need-Review                       │
                   (待审核)                             │
                           │                           │
              ┌────────────┼────────────┐              │
              ▼            ▼            ▼              │
         审核通过     发现依赖阻塞    审核不通过          │
              │            │            │              │
              ▼            ▼            │              │
       ⑧ Task-Done   ⑥ Blocked ───────┘  (退回④)      │
       (已完成)       (被阻塞)                          │
                           │                           │
                           │ 依赖解除                   │
                           ▼                           │
                    ⑦ Task-Reviewed                    │
                    (审核通过待合并)                     │
                           │                           │
                           ▼                           │

```

### 3.2 每个状态的详细定义

| 编号  | 状态名     | 英文          | 定义             | 进入条件       | 退出条件               | 责任人      |
|-----|---------|-------------|----------------|------------|--------------------|----------|
| ①   | 想法待完善   | Draft       | PM 的初始想法，信息不完整 | PM 创建需求    | 评审通过 → ②           | PM       |
| ②   | 想法      | Idea        | 经过内部评审的需求，描述清晰 | 评审通过       | 拆解为任务 → ③；需要补充 → ① | PM + 评审组 |
| ③   | 待执行     | Task-TODO   | 已拆解为可执行任务，等待分配 | PM 将想法拆解   | 分配并开始执行 → ④        | PM / TL  |
| ④   | 进行中     | In-Progress | 任务正在被执行        | 分配给执行者     | 执行完成 → ⑤           | 执行者      |
| ④-a | AI 执行中  | In-AI       | AI Agent 正在处理  | Agent 接收任务 | Agent 输出结果 → ⑤     | AI Agent |
| ④-b | 人工执行中   | In-Human    | 人工正在处理         | 分配给具体人     | 人工标记完成 → ⑤         | 开发者      |
| ⑤   | 待审核     | Need-Review | 执行结果等待审核       | 执行者提交产出    | 审核通过 → ⑦/⑧；不通过 → ④ | 审核者      |
| ⑥   | 被阻塞     | Blocked     | 因依赖未满足而无法推进    | 存在未完成的前置任务 | 依赖解除 → ④ 或 ⑤       | PM       |
| ⑦   | 审核通过待合并 | Reviewed    | 审核通过，等待部署/上线   | 审核者通过      | 上线完成 → ⑧           | TL / 运维  |
| ⑧   | 已完成     | Done        | 任务完成并上线        | 所有验收通过     | —                  | —        |

### 3.3 状态流转规则

```yaml
transitions:
  - from: Draft
    to: Idea
    trigger: 评审会议通过
    action: 自动通知 PM，需求进入候选池

  - from: Idea
    to: Draft
    trigger: PM 主动退回 / 评审要求补充
    action: 记录退回原因

  - from: Idea
    to: Task-TODO
    trigger: PM 拆解为具体任务
    action: 创建子任务，关联原始需求

  - from: Task-TODO
    to: In-Progress
    trigger: Agent/人工 接受任务
    action: 开始计时，通知 PM

  - from: In-Progress
    to: Need-Review
    trigger: 执行者提交产出物
    action: 自动分配审核人

  - from: Need-Review
    to: Blocked
    trigger: 审核发现依赖未满足
    action: 创建依赖关系，通知依赖任务负责人

  - from: Need-Review
    to: In-Progress
    trigger: 审核不通过
    action: 附带审核意见，退回执行者

  - from: Need-Review
    to: Reviewed
    trigger: 审核通过
    action: 进入发布队列

  - from: Blocked
    to: In-Progress
    trigger: 依赖任务完成
    action: 自动通知被阻塞任务的执行者

  - from: Reviewed
    to: Done
    trigger: 部署/上线完成
    action: 归档，生成完成报告
```

## 四、角色定义

### 4.1 角色矩阵

| 角色            | 职责               | 权限                |
|---------------|------------------|-------------------|
| **PM（产品经理）**  | 提交需求、拆解任务、追踪进度   | 创建/编辑需求、分配任务、查看全部 |
| **TL（技术负责人）** | 技术方案评审、审核产出、决定上线 | 审核任务、管理依赖、发布审批    |
| **开发者**       | 执行具体技术任务         | 接受任务、提交产出、标记完成    |
| **审核者**       | 验证产出质量           | 审核/退回任务、添加审核意见    |
| **AI Agent**  | 自动执行标准化任务        | 接收任务、输出结果、请求人工介入  |

### 4.2 AI Agent 的角色定位

AI Agent 在平台中不是"工具"，而是**一等公民执行者**：

- Agent 和人类开发者在同一个看板上，以相同的方式接收任务
- Agent 有自己的"能力标签"，例如：`规则代码生成`、`数据分析`、`测试用例编写`
- 当 Agent 遇到无法处理的情况时，它会自动将任务流转给人工（④-a → ④-b）
- Agent 的每次执行都有完整的日志记录，供审核者追溯

**需求卡片结构：**


### 5.3 任务拆解（Task Breakdown）

PM 可以将一个需求拆解为多个任务，并定义依赖关系：

```
需求 REQ-0042: 新增黑名单规则
│
├── 任务 T-001: 规则逻辑设计        [人工] → 无依赖
├── 任务 T-002: 数据表结构变更        [人工] → 依赖 T-001
├── 任务 T-003: 规则代码编写          [AI Agent] → 依赖 T-001
├── 任务 T-004: 测试用例编写          [AI Agent] → 依赖 T-003
├── 任务 T-005: 联调测试              [人工] → 依赖 T-002, T-004
└── 任务 T-006: 上线部署              [人工] → 依赖 T-005
```

**依赖类型：**

| 类型          | 含义          | 阻塞规则                              |
|-------------|-------------|-----------------------------------|
| `blocks`    | 前置任务完成后才能开始 | 上游 Done 后下游才能从 TODO → In-Progress |
| `related`   | 关联但不阻塞      | 仅展示关联关系，不阻止流转                     |
| `duplicate` | 重复任务        | 标记后合并                             |

### 5.4 Agent 调度（Agent Dispatch）

**Agent 能力注册：**

每个 Agent 在平台注册时声明自己的能力：

```yaml
agent:
  name: "规则代码生成Agent"
  id: agent-rule-coder
  capabilities:
    - type: "code_generation"
      domain: "风控规则"
      languages: [ "Python", "SQL" ]
      confidence: 0.85
    - type: "test_generation"
      domain: "单元测试"
      confidence: 0.90
  escalation:
    condition: "confidence < 0.6 OR human_feedback_required"
    action: "transfer_to_human"
    notify: [ "tl@team.com" ]
```

**自动分配逻辑：**

```
任务进入 Task-TODO
    │
    ▼
匹配 Agent 能力标签
    │
    ├── 匹配成功且置信度高 → 分配给 Agent（→ ④-a）
    ├── 匹配成功但置信度低 → Agent 先处理，人工复核
    └── 无匹配 Agent → 分配给人工（→ ④-b）
```

**Agent 执行过程中的交互：**

```
Agent 在执行过程中可能触发以下事件：

1. [完成] → 正常输出结果，进入审核
2. [求助] → Agent 遇到歧义，向 PM/TL 发起提问
3. [升级] → Agent 判断超出能力范围，自动转人工
4. [并行] → Agent 调用子 Agent 处理子任务
5. [重试] → 审核不通过后，Agent 尝试修正
```

### 5.5 审核系统（Review）

**审核流程：**

```
执行者提交产出
    │
    ▼
系统自动分配审核人（基于规则）
    │
    ├── 规则1：代码类任务 → TL 审核
    ├── 规则2：规则配置类 → 风控专家审核
    └── 规则3：AI 产出 → 人工 + AI 交叉验证
    │
    ▼
审核人操作
    │
    ├── ✅ 通过 → 进入下一阶段
    ├── ❌ 不通过 → 退回并附带意见
    └── ⚠️ 有条件通过 → 通过但标记待改进项
```

**审核清单（Checklist）：**

审核者不是"随便看看"，而是按照预定义的清单逐项检查：

- [ ] 产出物是否满足需求中的验收标准
- [ ] 代码/规则是否符合团队规范
- [ ] 是否有潜在的风控漏洞
- [ ] AI 产出是否有幻觉/错误
- [ ] 测试覆盖是否充分

### 5.6 阻塞管理（Block Management）

**阻塞可视化：**

```
依赖关系图：

T-001 ──► T-002 ──► T-005 ──► T-006
  │                  ▲
  └──► T-003 ──► T-004

当前状态：
  T-001 ✅ Done
  T-002 ✅ Done
  T-003 🔄 In-Progress (AI Agent 60%)
  T-004 ⏳ Blocked (等待 T-003)
  T-005 ⏳ Blocked (等待 T-004)
  T-006 ⏳ Blocked (等待 T-005)
```

**阻塞预警：**

- 当一个任务被阻塞超过 N 天，自动通知 PM
- 当阻塞形成环形依赖，系统报警
- PM 可以手动解除阻塞（需要填写原因）

## 六、自动化规则引擎

### 6.1 触发器 + 条件 + 动作

平台内置一个轻量的规则引擎，团队可以自定义自动化逻辑：

```yaml
rules:
  # 规则1：AI Agent 产出自动触发审核
  - name: "AI产出自动进审核"
    trigger: "task.status changed to Need-Review"
    condition: "task.executor.type == 'AI'"
    action:
      - assign_reviewer(role: "风控专家")
      - add_checklist(template: "AI产出审核清单")
      - notify(reviewer, method: "钉钉")

  # 规则2：超时自动升级
  - name: "任务超时预警"
    trigger: "task.in_progress_duration > 3_days"
    condition: "task.priority == 'P1'"
    action:
      - notify(pm, message: "P1任务已进行3天未完成")
      - escalate(to: "TL")

  # 规则3：依赖解除自动通知
  - name: "阻塞解除通知"
    trigger: "blocked_task.dependency.status changed to Done"
    action:
      - unblock(task)
      - notify(executor, message: "你的任务已解除阻塞")
      - move_status(task, to: "In-Progress")

  # 规则4：每日站会报告
  - name: "每日站会报告"
    trigger: "cron: every weekday 9:30"
    action:
      - generate_report(type: "standup")
      - send_to(group: "风控团队群")
```

### 6.2 预置模板

平台为风控场景预置常用工作流模板：

| 模板名         | 适用场景      | 特色                |
|-------------|-----------|-------------------|
| **风控规则上线流** | 新规则从设计到上线 | 自动关联数据表变更、测试、灰度发布 |
| **紧急需求响应流** | 突发风控事件    | 跳过非必要环节，自动加急      |

| **合规审查流**   | 监管合规类需求   | 强制审核节点，保留完整审计链    |

### 7.2 关键指标

| 指标             | 计算方式              | 意义           |
|----------------|-------------------|--------------|
| **需求周期**       | Done时间 - Draft时间  | 一个需求从提出到上线多久 |
| **任务周期**       | Done时间 - TODO时间   | 一个任务执行多久     |
| **阻塞率**        | Blocked任务数 / 总任务数 | 团队协作效率       |
| **Agent 自动化率** | AI完成任务数 / 总完成任务数  | AI 替代人工的程度   |
| **审核通过率**      | 一次通过数 / 总审核数      | 产出质量         |
| **需求吞吐量**      | 每周完成需求数           | 团队产能         |

## 九、技术选型建议

| 层级       | 建议                    | 理由                               |
|----------|-----------------------|----------------------------------|
| **前端**   | React + DnD Kit       | 看板拖拽体验好，组件生态丰富                   |
| **后端**   | Go / Python (FastAPI) | Go 高性能适合状态机引擎；Python 适合 Agent 集成 |
| **状态机**  | 自研轻量引擎 或 XState       | 状态流转是核心，需要可控                     |
| **数据库**  | PostgreSQL + Redis    | PG 存任务和关系；Redis 缓存看板状态           |
| **消息队列** | RabbitMQ / Kafka      | Agent 异步执行、事件通知                  |
| **实时通信** | WebSocket             | 看板实时更新                           |
